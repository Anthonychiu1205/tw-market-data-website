# TWMD 官網工程標準憲章 · 2026-07-14

**緣起**:2026-07-13~14 的密集修復挖出一串官網 bug(死方案表含 RPM 10、developer 語意倒置、creditsReference 過時 NT$ 定價、hero 捏造 2330 K線/報價、demo 假指數 23,481)。**它們不是隨機的,是系統性缺口的症狀。** 本憲章把教訓寫成前向規則。與 API repo 的 `docs/ENGINEERING_STANDARDS.md` 同源。

---

## 1. 單一真相源(SSOT)· 零重複 · 零死碼
- **定價唯一源 = `plans.ts`**;方案/限額/狀態不得在別處有第二份表。
- 狀態/覆蓋對齊 **llms.txt / coverage-facts SSOT**;不逐格編造覆蓋數字。
- **死碼(grep 零引用)= 刪,不維護**。已發生的事故:pricingPlans/pricingMatrix/creditsReference/hero-market-preview 都是死碼被當活文案改過(RPM 10、假 2330 報價藏在裡面)。

## 2. 零捏造對外數字(硬規則)
- 對「逐值對官方對帳」的公司,**任何假數字 = 最傷的自打臉**。
- 官網任何對外數字(價格/K線/指數/統計/覆蓋)**必須來自真實源,否則整塊不顯示**。
- **禁止假數字貼在真實 ticker 上**(如 2330 台積電)。demo/placeholder 視同地雷:寧可空、不可假。

## 3. 顯示走真源,非硬編碼
- 首頁/docs/定價的統計與指數**接 SSOT/API 即時值**,禁硬編碼過時數字(37,196 那類)。
- 沒真值就誠實不顯示,等真源;不編造。

## 4. 改/刪前先確認 render
- 改或刪任何 config/元件前,**grep 全 repo 確認引用**(排除 node_modules/.next/.git)。
- 死碼(零 render)→ 刪;活碼 → 接真源。**不改版面設計、不加框框**(外科手術式修正)。

## 5. 改一個數字前先問「有幾個源」
- 發現不一致 → **找 SSOT、刪其餘**,不是補一個值。根因修,非症狀修(RPM 10→60 一路挖出整代死表的教訓)。

- [ ] 沒有新的重複真相源;沒留死碼(新加零引用要說明用途)
- [ ] 沒有捏造對外數字;假不了就不顯示
- [ ] 顯示數字走真源(SSOT/API),非硬編碼過時值
- [ ] 改/刪前 grep 確認引用;版面/文案零意外變動
- [ ] Vercel preview 綠再 merge

**一句話:單一真相源、零捏造、顯示走真源、根因修。**
